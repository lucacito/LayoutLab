# Phase 6a — Programmatic Taxonomy Landing Pages — Design

**Status:** Approved (brainstorm) — 2026-06-28
**Roadmap:** CLAUDE.md §7/§13/§19 Phase 6 (programmatic SEO), first slice.
**Predecessor:** Phase 5c (transactional emails) — complete, tagged `phase-5c-complete`.

---

## Goal

Ship the programmatic-SEO surface: a landing page for every taxonomy axis value
(`/type/hero`, `/niche/saas`, `/style/minimal`, `/color/blue`) with AI-written
intro copy, a filtered layout grid, and `ItemList` + `BreadcrumbList` JSON-LD,
all enumerated from data and added to the sitemap.

---

## Key decisions (resolved in brainstorm)

1. **Axis-prefixed routes:** `/type/[type]`, `/niche/[niche]`, `/style/[style]`,
   `/color/[color]` — avoids colliding with `/layouts/[slug]`, consistent across
   the four axes. Single-axis pages now; combos deferred.
2. **AI-generated copy, stored per page** — generated by a script (reusing the
   pipeline's `LlmClient`, the `claude -p` subscription backend) and stored, with a
   **deterministic templated fallback** so every page renders before generation has
   run. The four axes are `type`, `niche`, `style`, `color` (the real filterable
   axes; `AXIS_VALUES` is the canonical list).

### Why a dedicated `taxonomy_pages` table (not `tags`)

The existing `tags` table's `tagAxis` enum is `['type','niche','style','feature']`
(no `color`) and it backs per-layout tagging (`layout_tags`). Landing-page copy is
a different concern keyed by `(axis, value)` across all four real axes including
`color`. A small dedicated table keeps them separate and unconstrained.

---

## Architecture & data flow

```
Build/runtime:
  /type/[type] (+ /niche, /style, /color) — generateStaticParams from AXIS_VALUES[axis]
    │  validate param ∈ AXIS_VALUES[axis]            → notFound() if not
    │  listLayouts({ [axis]: [value], … })           → the filtered, published grid
    │  getTaxonomyCopy(axis, value)  ?? taxonomyFallbackCopy(axis, value, count)
    ▼  render: intro copy + LayoutCard grid + ItemList JSON-LD + BreadcrumbList JSON-LD
       generateMetadata: title/description from the (stored-or-fallback) copy

Offline (one-time / on-demand):
  npm run seo:copy → for each (axis, value): if no stored intro,
    LlmClient.complete(prompt) → { intro, metaTitle, metaDescription } → upsertTaxonomyCopy
    (idempotent: skip already-generated; resumable; budget-capped)

Sitemap: add every (axis, value) URL alongside layouts + packs.
```

## Components / units

### 1. Schema + store — `db/schema.ts`, `lib/seo/taxonomy.ts`
- New table `taxonomy_pages`: `axis` text, `value` text, `intro` text, `metaTitle`
  text, `metaDescription` text, `updatedAt` timestamp; unique `(axis, value)`.
  Migration via drizzle-kit.
- `TaxonomyAxis = 'type' | 'niche' | 'style' | 'color'`.
- `getTaxonomyCopy(axis, value): Promise<{ intro: string; metaTitle: string; metaDescription: string } | null>`.
- `upsertTaxonomyCopy(axis, value, copy): Promise<void>` (onConflict update on `(axis,value)`).

### 2. Templated fallback — `lib/seo/taxonomy-copy.ts`
- `axisLabel(axis, value): string` — humanizes (`real_estate` → "Real Estate",
  `full_landing` → "Full Landing Page").
- `taxonomyFallbackCopy(axis, value, count): { intro; metaTitle; metaDescription }`
  — deterministic, readable, count-aware copy. Pure, unit-tested.

### 3. AI copy generation — `pipeline/seo-copy.ts` + `npm run seo:copy`
- Reuses the pipeline `LlmClient` (constructed as in `pipeline/index.ts`/`run.ts`).
- For each `(axis, value)` across the four axes: if `getTaxonomyCopy` already has an
  intro, skip (idempotent/resumable); else prompt for `{ intro, metaTitle,
  metaDescription }`, parse (reuse `extractJson`), clamp lengths, `upsertTaxonomyCopy`.
  Budget-capped per call. CLI logs progress; never crashes the whole run on one
  failure (logs + continues).
- Unit-tested with a mocked `LlmClient` (no real CLI): asserts skip-when-present,
  upsert-on-success, continue-on-parse-failure.

### 4. Landing pages — `app/(catalog)/{type,niche,style,color}/[value]/page.tsx` + `components/TaxonomyLanding.tsx`
- Four thin route files, each: `generateStaticParams` from `AXIS_VALUES[axis]`;
  `generateMetadata` (stored-or-fallback copy → title/description, canonical);
  the page validates the param (`notFound()` if not in `AXIS_VALUES[axis]`),
  fetches layouts + copy, and renders `<TaxonomyLanding axis value layouts copy />`.
- `TaxonomyLanding` (shared, server): breadcrumb, H1 + intro, the `LayoutCard`
  grid (empty-state when none), and injects `itemListJsonLd(layouts)` +
  `breadcrumbJsonLd([...])` via the existing `lib/seo/jsonld` helpers and a
  `<script type="application/ld+json">`.

### 5. Sitemap — `lib/seo/sitemap.ts`, `app/sitemap.ts`
- Extend `sitemapEntries` to also emit the taxonomy URLs (every `AXIS_VALUES`
  value × 4 axes). `app/sitemap.ts` passes them (static list — no DB needed for the
  taxonomy rows).

## Error handling

- **Invalid axis value** in the URL → `notFound()` (404), not a crash.
- **No layouts for a value** (valid but empty) → the page still renders (intro +
  an empty-state message); it stays in the sitemap. (Thin pages are acceptable; the
  catalog grows.)
- **No stored copy** → the deterministic templated fallback renders; the page is
  fully functional before `seo:copy` runs.
- **Generation:** a per-value LLM/parse failure is logged and skipped; the script
  continues and is re-runnable (idempotent).

---

## Testing strategy (TDD)

- **Pure unit:** `axisLabel` (humanization incl. `real_estate`, `full_landing`);
  `taxonomyFallbackCopy` (count-aware, non-empty title/description/intro);
  `sitemapEntries` includes the taxonomy URLs (count = 4 axes × their values, plus
  layouts + packs).
- **Generation unit:** `seo-copy` core with a mocked `LlmClient` — skips when copy
  exists, upserts on valid JSON, continues on a parse failure (no throw).
- **Component:** `TaxonomyLanding` renders the H1/intro, the grid (or empty state),
  and emits both JSON-LD blocks containing the right `@type` (`ItemList`,
  `BreadcrumbList`).
- **Route:** an invalid axis value → `notFound()` (assert the page calls it / 404
  branch).
- **Integration (gated on `POSTGRES_URL`):** `getTaxonomyCopy`/`upsertTaxonomyCopy`
  round-trip; a landing page lists only published layouts for its value.
- **Build:** the four routes + the sitemap compile; `generateStaticParams` yields
  all axis values.
- **Manual:** `/style/minimal` shows fallback copy + the minimal layouts + valid
  JSON-LD (Rich Results check); run `npm run seo:copy` (local claude CLI) → the page
  now shows AI copy; `/type/not-a-type` → 404; the sitemap lists the taxonomy URLs.

events, and the perf pass.
