# Phase 3.5 ‚Äî Storefront Design System & Frontend ‚Äî Design

**Status:** Approved (brainstorm) ‚Äî 2026-06-28
**Roadmap:** Inserted before Phase 4 (commerce) so checkout/account inherit the system.
**Predecessor:** Phase 3a (generation pipeline) ‚Äî complete, tagged `phase-3a-complete`

---

## Goal

Give the storefront a real, cohesive visual identity and the site chrome it
currently lacks. Today the pages render real data but have no header/nav/footer,
no design tokens, and a Phase-0 placeholder home page ‚Äî so the app *reads* as
broken even though it works. This phase builds a themeable design system from the
user's Calendly-style brand brief and applies it across the public catalog, a new
landing page, and (lightly) the admin UI. **Visual layer only ‚Äî no data-model,
commerce, or pipeline changes.**

Definition of done: the home page, browse, layout & pack detail, and admin pages
render in the brand system, responsive and accessible (WCAG-AA), the existing
test suite stays green, logic-bearing UI primitives are unit-tested, and key
pages are visually verified on localhost.

## Brand brief (source of truth ‚Äî from the user)

**Theme:** modern productivity SaaS ‚Äî simple, trustworthy, human, efficient,
premium, calm technology. **"The interface is the hero"**: clean canvas, product
screenshots and UI cards as floating objects, minimal decoration. Reference:
Calendly.

**Color tokens:**

| Token | Hex | Use |
|---|---|---|
| `navy` | `#0B3558` | headlines, logo, high-emphasis text, nav (never pure black for big headings) |
| `blue` (Action Blue) | `#006BFF` | primary buttons, links, active/interactive |
| `paper` | `#FFFFFF` | cards, panels, mockups |
| `mist` | `#F8F9FB` | page backgrounds, section separation |
| `fog` | `#E7EDF6` | input bg, secondary cards, badges |
| `slate` / `muted` | `#476788` | paragraph + secondary text, metadata |
| `text` | `#0A0A0A` | body text (near-black) |
| `border` | `#D4E0ED` | borders, secondary-button outline |
| decorative | `gradient-pink #E55CFF`, `gradient-purple #8247F5`, `gradient-yellow #FFA600`, `gradient-cyan #0099FF` | background blobs/illustration ONLY ‚Äî never buttons/text/UI |

**Type:** geometric sans (Gilroy) with fallback **Inter** / Manrope / DM Sans.
Scale: Display 80/700/1.1 ¬∑ H1 68/700/1.2 ¬∑ H2 50/700/1.2 ¬∑ H3 38/700/1.25 ¬∑
Section 28/600/1.4 ¬∑ Large body 20/500/1.6 ¬∑ Body 18/400/1.6 ¬∑ Small 14/400/1.5.

**Layout:** container `1200px` centered; 8px spacing base
(8/16/24/32/48/64/80/120); section spacing 80‚Äì120px.

**Buttons:** primary ‚Äî 48px height, 16√ó24 padding, `blue` bg, white text, radius
`4px`, NOT pill. Secondary ‚Äî white bg, `1px solid #D4E0ED`, navy text, radius 4px.

**Cards:** paper bg, `1px solid #D4E0ED`, radius `16px`, padding `24px`, soft
blue-tinted shadow `0 30px 50px rgba(71,103,136,.08)` (avoid heavy black shadows).

**Signature decoration:** floating gradient shapes (blurred, low opacity,
gradient) behind product UI.

**Icons:** 24px simple line icons, navy or blue, no filled illustrations.

**Imagery:** product screenshots / UI cards / browser mockups ‚Äî NO stock photos,
generic smiling people, or abstract 3D. (Our layout screenshots fit this exactly.)

**Voice:** confident, clear, helpful ("Save time scheduling meetings", not
"revolutionary AI-powered synergy").

### Resolved interpretive calls
1. **Font: ship Inter** (the user's listed fallback) via `next/font` ‚Äî Gilroy is
   proprietary; the type system is structured so Gilroy drops in later by

2. **Admin gets a light restyle** (tokens + Button/Card), not a deep redesign ‚Äî

3. **Brand name stays `LayoutLab`** (placeholder) ‚Äî wordmark only, no logo asset.

## Architecture & approach

A token-first design system: brand values live once in `tailwind.config.ts`
(theme) + a few CSS custom properties in `globals.css`; small presentational
primitives in `components/ui/` consume the tokens; pages compose primitives.
Refactor the existing ad-hoc Tailwind in current components onto the primitives
rather than leaving two styling vocabularies.

```
tailwind.config.ts + globals.css   ‚Üê brand tokens (color/type/spacing/radius/shadow/font)
        ‚îÇ
components/ui/*  (Button, Card, Badge, Container, SectionTitle, IconFeature,
        ‚îÇ         GradientBlob, Wordmark)  ‚Üê consume tokens, no business logic
        ‚ñº
app/layout.tsx (Header + Footer chrome) ‚Üí every page
app/(marketing)/page.tsx  (new landing)  ¬∑ (catalog) pages restyled ¬∑ admin lightly restyled
```

No new data flow. Pages keep their existing server queries; only presentation
changes. Catalog visibility, ingest, admin gating, and the pipeline are
untouched.

## Components / units

### 1. Tokens ‚Äî `tailwind.config.ts`, `app/globals.css`, font wiring
- Tailwind `theme.extend`: `colors` (all brand tokens), `borderRadius`
  (`button: 4px`, `card: 16px`), `maxWidth.container: 1200px`, `boxShadow.soft`,
  `fontFamily.sans` (Inter var), spacing already 4px-based (8px steps via 2,4,6‚Ä¶).
- `globals.css`: base element styles (body bg `mist`, text color, antialiasing),
  the type-scale utility classes (or a small set of `@layer` components for
  `.text-display/.text-h1/...`), and `prefers-reduced-motion` handling for blobs.
- `app/layout.tsx`: load Inter via `next/font/google` and apply the CSS variable.

### 2. UI primitives ‚Äî `components/ui/`
Each is a small, prop-driven, server-renderable component (client only when it
needs interactivity):
- `Button({ variant: 'primary'|'secondary', as?: 'a'|'button', ... })`
- `Card`, `Badge`, `Container`, `SectionTitle`, `IconFeature({ icon, title, body })`,
  `GradientBlob({ palette })` (decorative, `aria-hidden`), `Wordmark` (navy
  "LayoutLab" linking home).
- Logic-bearing ones (`Button` variant‚Üíclass mapping, `Wordmark`) are unit-tested.

### 3. Global chrome ‚Äî `components/site/Header.tsx`, `Footer.tsx`, `MobileNav.tsx`
- `Header`: sticky, paper bg with bottom border, `Wordmark`, primary nav
  (Browse ¬∑ Pricing ¬∑ License), `Sign in` link + a primary `Button` CTA; collapses
  to a `MobileNav` (client) below `md`.
- `Footer`: brand blurb, link columns (Catalog, Company, Legal), copyright.
- Wired into `app/layout.tsx` so all routes get chrome. (Admin pages may render a
  slim variant or reuse the same header ‚Äî decided in the plan.)

### 4. Landing page ‚Äî `app/(marketing)/page.tsx` (moves `/` off the stub)
Sections per the brand template, fed by existing catalog queries:
- **Hero** (two-column): headline + supporting copy + primary CTA (‚Üí `/browse`) /
  a floating layout-screenshot `Card` with `GradientBlob`s behind.
- **Trust strip** ("Validated, import-ready Divi 5 layouts" + counts).
- **Featured packs** (real published packs via `listPacks`).
- **Feature blocks** (alternating `IconFeature` rows ‚Äî validated, screenshots,
  one-click import).
- **Navy CTA section** (full-width navy bg, white headline, blue `Button`).
Uses real screenshots (placeholder picsum today; real ones after Phase 3b).

### 5. Restyle catalog ‚Äî `app/(catalog)/*` + `components/*`
- `LayoutCard`/`PackCard` ‚Üí `Card` system (16px radius, soft shadow, fog hover,
  navy title, slate meta).
- `/browse`: styled facet sidebar (fog inputs, navy legends), `SearchSort`, grid;
  styled empty state.
- `/layouts/[slug]`: navy headings, slate body, gallery + lightbox polished,
  "appears in packs" as cards.
- `/packs/[slug]`: price display, primary/secondary `Button` CTAs (stubbed to
  `/pricing`), included-layouts grid.
- `Breadcrumbs` restyled to tokens.

### 6. Light admin restyle ‚Äî `app/admin/*`, `components/admin/QueueTable.tsx`
Apply `Container`, `Button`, `Card`, and tokens so admin is visually consistent;
keep its structure and behavior unchanged.

### 7. States, responsive, a11y
Hover/focus-visible rings (blue), empty + loading states, mobile‚Üídesktop
breakpoints (single-column hero/grids on small screens), AA contrast, semantic
landmarks (`header`/`nav`/`main`/`footer`), `aria-hidden` on decorative blobs,
`prefers-reduced-motion`.

## Error handling

Pure presentation, so little runtime risk. Guard the data-fed sections: the
landing page's featured packs/layouts render an empty-but-valid state when the DB
returns none (or is unreachable in a non-DB build) rather than throwing ‚Äî the
home page must never 500 on missing data. Images keep `next/image` fallback
behavior. No new external calls.

---

## Testing strategy

- **Unit (jsdom):** logic-bearing primitives ‚Äî `Button` renders the correct
  variant classes and the right element (`a` vs `button`); `Wordmark` links home;
  `IconFeature` renders title/body. (Pure visual components aren't unit-tested.)
- **Regression:** the full existing suite (95 tests) stays green ‚Äî restyled
  components must keep their tested behavior (e.g. `LayoutCard` still links to
  `/layouts/<slug>` and shows the title; `QueueTable` still renders rows +
  Approve/Reject).
- **Build:** `npm run build` compiles all routes (incl. the new landing under the
  `(marketing)` group).
- **Visual verification:** run the dev server against the seeded local DB and
  screenshot the home page, `/browse`, a layout detail, a pack detail, and
  `/admin/queue` ‚Äî confirm chrome, tokens, responsive, and the gradient-blob hero
  render as intended. (This is the primary acceptance for a visual phase.)
